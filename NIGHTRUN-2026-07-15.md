## 1. R8 — the exam was indicting the product for succeeding 🔴

`report` printed DEGRADED. R8 said **12/13 snapshots regressed**. Both were wrong.

```python

```

Measured on the live store:

```
dropped=17  added=17  stale=16
stale by reason: {'killed': 15, 'decayed': 1}
snapshots where a baseline memory dropped and NOTHING was retired: 1
```

**Sixteen of seventeen missing baseline memories were missing because Helicon had
killed them as rot.** Retrieval filters killed+superseded at the source, so they
*cannot* come back and *must not*. A better memory took each vacated slot
(`added=17`). That is the loop working end to end, reported as degradation — and
`report` printed DEGRADED off the same count, so the one command a judge runs in
thirty seconds indicted the product for doing its job.

A regression is now only what it always should have been: **a memory that is
still live and no longer retrieved.** The exam can't tell whether a better memory
displaced it, so it files it and a human rules. That's the loop.

**12/13 → 1/13**, and the 1 is real: two live memories (`topic_search.py`,
`search.py`) that used to be retrieved for "Search" and no longer are.

### It also could not reproduce its own verdict

Three identical runs on identical data: **11/13, 12/13, 11/13.**

Cause: retrieval calls **qwen3-rerank, a remote model**, on every query. Proven
non-deterministic on fixed input:

```
call 1: (1, 5, 7, 6, 2)
call 2: (1, 5, 7, 6, 2)
call 3: (6, 5, 1, 7, 4)     <- same query, same documents
```

And `except Exception: return None` silently fell back to the hybrid order, so a
timeout quietly changed the ranking strategy with **no error anywhere**. The
agent saw different context run to run and nothing said so.

Fixed three ways: `np.argsort` defaults to quicksort (not stable) so equal scores
came back arbitrarily ordered and near-duplicate cubes tie constantly — now
`(-score, id)`, costing **+2ms per query** on 7k vectors. The RRF fusion sort had
the same latent tie. And rerank is memoized on its actual inputs, **in-process
and durably in `qwen_cache`** — because every CLI invocation is a fresh process,
and `helicon rot` three times still gave `CLEAN / CLEAN / ROT FOUND`, which is
exactly what a judge would hit.

Now identical across processes. **`rot` went 21s → 9.7s.**

> **Honest limit:** caching makes the verdict reproducible; it does not make the
> reranker deterministic. It freezes one sample of a non-deterministic function.
> A different day would have frozen a different sample.

`b3e3c69` · 9 tests, reverting the definition fails 8 · **nothing covered
`check_snapshot` before this, which is why it shipped.**

## 2. #361 — recording a ruling manufactured the evidence it was violated 🔴

Two independent faults, both proven live.

**The parser could not read its own canonical format.**

```
_intervals_in('09-11..09-13')  ->  []          <- what _iv_label EMITS
_intervals_in('Sep 11-13')     ->  [('09-11','09-13')]
```

`MM-DD..MM-DD` is the format every ruling, resolution and GOLDEN_RULE is written
in. Resolutions are parsed by `_parse_label`, which understands `..`. Cube
*content* was parsed only by the prose patterns. **Two parsers, two grammars, one
format.** So this line:

```
#322 Itai wedding = **09-11..09-13** (the "Aug 14-22" cube was mislabeled)
```

asserted **only the corpse it quoted**, never the truth it stated.

**The never-twice guard ate the correction.** It re-alarms on any
post-resolution cube asserting the dead value — correctly, that is the
guarantee. But the memory that *records* a ruling is written moments after it
and must name the dead value to say what was wrong. **The act of ruling created
the evidence that the ruling had been violated.** Every ruling planted a future
false alarm, inflating R1 forever and discrediting the one exam that has to stay
trustworthy.

The same shape catches enumeration: `Italy (Aug 14-22) and Itai's wedding, Sweden
(Sep 11-13)` is one line about two trips, and keyword proximity grabs both dates.

**Live: 1 → 0 dated-fact conflicts. R1 9 → 8.** #361 dismissed with its cause;
precedents **19 → 20** — the loop closing on a finding the loop had manufactured.

## Adversarial pass — it found four real defects, three of them mine

An independent reviewer was asked to REFUTE all three fixes, defaulting to
"broken". It could not construct the R8 masking case (the claim I most needed
attacked) and it withdrew my worst-case cost worry with a measurement:
`code_refs` is **1.5s across the 27 real repos, not 9 minutes**. It also verified
every falsifiability claim in my commit messages is exactly accurate.

Then it found this, and it was right:

**D1 (CRITICAL) — `debac36` did NOT close the hole it claimed to close.** The
`(cube, line)` re-key was defeated by a dedupe two functions away.
`extract_assertions` keeps the FIRST occurrence of `(person, topic, interval)`,
and a correction line quotes both poles — so when the ruling came first, the
genuine re-assertion further down inherited the correction's line and was
excluded with it:

```
RULING line 1, ROT line 5  ->  SILENT   <- rot missed
ROT line 1, RULING line 5  ->  fires
```

Order-dependent, and **the natural order was the broken one**. My regression test
passed only because its truth line did not quote the corpse — the one shape with
no dedupe collision. Fixed at the dedupe: when several lines assert the same
interval, keep the one that asserts it ALONE. A solo line is never a correction,
because a correction names what it replaces. `a405c69`

**D6 — `rerank_health()` had zero callers.** My commit said it "makes the silent
fallback visible". Nothing called it. **Dead code shipped as the fix for
silence** — the exact sin this night was about. Worse, the counter was useless by
construction: it tallied an in-memory list, and every CLI invocation is a fresh
process, so it was always zero when asked. Same lesson as the nightly: liveness
is a state you assert, not an event you tally. It now probes the reranker and
prints in `doctor`, healthy or not. `a405c69`

**D4 — the rename-aware rule hid one of the three identical outages in the file
I cited as proof.** `seed/route.ts` lines 11, 12, 13 all carry `agentId:"relay"`.
Only 11 and 13 were reported; line 12 vanished because its product copy reads
*"What favour would you ask someone nearby"* — prose matching `\bfavour\b`. Lines
11 and 13 survived by luck ("Favours"/"favourite" don't match `\b`). The bug was
asymmetry: I required the OLD name to be a quoted token and let the NEW name
match as a bare word, in a codebase whose domain vocabulary IS the new name.
Fixed: same rule both sides. `d651d09`

**D3 — `fossil` is computed, returned, and read by no verdict.** Confirmed: it
reaches only `rot.py`'s detail string. A fossil baseline, or an empty one, passes
forever with `live_overlap=1.0`. The test docstring literally says "silently
passing forever is how a dead test looks exactly like a healthy one" and then
does not wire the flag to anything. **Named, not fixed.**

## Failure inventory — what is still wrong

1. **R8 ignores `added` and `reordered` entirely.** `top_k=5`. A right memory
   ranked 5th instead of 1st still counts as retrieved, but an agent reading top-3
   never sees it. That failure mode is now invisible — **the most likely real gap
   in tonight's work**, so I measured it instead of leaving it a worry:

   ```
   total rank moves on the live store: 19  |  moves of 2+ places: 1
   the one 2-place move: "Mode Overview"  rank 4 -> 2   (it moved UP)
   ```

   Real in principle, **benign today**: every move is ≤2 places and mostly
   upward, which is what you'd expect when retired cubes vacate slots and
   everything shifts up. Nothing is currently being masked. The right shape is
   probably to report rank drift as informational without making it a verdict —
   a memory falling 1st → 5th deserves a look, but calling it "rot" is how R8 got
   into trouble in the first place.
2. **The rerank cache has no TTL or invalidation.** The key includes the
   documents, so a changed candidate set re-ranks live — but a frozen order
   outlives anything that should have changed the ranking without changing the
   candidates.
3. **Caching a non-deterministic function freezes an arbitrary sample.** Stated
   in the commit and in SUBMISSION.md rather than buried, but it is not a fix for
   the underlying non-determinism.
4. **R4's code arm flags docstring CONTINUATION lines** — `aliases.py:70` flags
   itself. `_COMMENT_RX` only matches a comment at line start. Multi-line
   docstrings need parsing, not a line regex.
5. **`new_rx` may over-suppress.** `const id = legacy ? "relay" : "favour";`
   names both names, so it reads as rename-aware — but it IS an outage. Not
   constructed, not disproven.
6. **R4's code arm shells `git ls-files` per repo on every `rot`.** 27 repos, 20s
   timeout each = a 9-minute worst case. Currently 9.7s total, so it is fine
   today and fragile in principle.
7. **The 8 remaining R1 conflicts are the claim selector** (`claims.py`), a
   different mechanism. GOLDEN_RULES already carries dismissals of the same
   both-poles shape there, so it is probably the next instance of #361 — not
   investigated.
8. **`review --terminals` is still thin** (the findings doc's closing point, and
   what Oscar actually asked for). Untouched: the queue was R8 first.
