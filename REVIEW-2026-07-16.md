## 2. The guard fix (the real one) — ruled facts now BIND at write time

**Before** (both clean — the bug): `guard.py` only checked aliases (renames) and identity rulings (`audit_type='identity'`). It never read factual claim rulings, so a value a human explicitly ruled wrong sailed through.

```
$ helicon guard "4 hackathon wins"     →   ✓ clean   (WRONG: wins=9 was ruled, 4 marked wrong)
$ helicon guard "9 hackathon wins"     →   ✓ clean
```

**After** (fixed):

```
$ helicon guard "4 hackathon wins"
  BLOCKED — 1 ruling(s) contradict this output:
    [critical] wins for 'hackathon' was ruled '9', but this asserts '4' — ruled wrong (re-alarms if it returns).
        ↳ ruling #281 on 'hackathon' wins at 2026-07-06T08:58:48

$ helicon guard "9 hackathon wins"                                  →   ✓ clean
$ helicon guard "9 hackathon wins, 4 finalist placements, $115K"    →   ✓ clean  (no false positive)
```

**What changed** (`helicon/guard.py`):
- Added `_load_factual_resolutions(conn)` — mirrors `identity._load_identity_resolutions`. Reads `audit_log` rows where `audit_type='factual'` and `human_decision LIKE 'resolved:%'`, parses the ruled-true value (after `resolved:`) and the competing wrong values (from `all_dates`/`dates`/`value_a`/`value_b`), keyed by topic + subject.
- Added a third check in `guard_output()` that flags **critical** when the proposed text asserts a ruled-wrong value for that topic.

**Precision (the load-bearing detail):** finding #281's own canonical line is `9 hackathon wins, 4 finalist placements` — it contains a `4`. A naive "wrong value + topic co-present" check would flag the *true* line. The fix requires **adjacency**: the value must quantify the topic noun (`<value> [subject] <topic>`) or be an explicit assignment (`wins: 4`). In the true line the `4` quantifies "finalist", never reaches "wins", so it stays clean. A distinctiveness gate additionally catches dated/worded values (e.g. `birthday 07-13`) without reopening that trap. Covered by three new tests.

**Tests:** `tests/test_guard.py` +3 (`test_ruled_wrong_fact_is_blocked`, `test_reasserting_the_ruled_truth_fact_is_clean`, `test_ruled_fact_no_false_positive_on_canonical_line`). Full suite: **352 passed** (was 349), 0 regressions. The `helicon_guard` MCP tool picks this up automatically (it calls `guard_output`).

Why it matters: the submission's honesty thesis is "adjudication that binds." Until today the guard did not enforce the single most common ruling class (factual corrections). Now it does — the thesis is true, not aspirational.

## 3. Competition differentiation (for these judges)

The MemoryAgent field the judges will see most: **Mem0, Zep/Graphiti, Cognee** — memory *stores* with *automatic* contradiction handling. That automation is the gap.

- Mem0: LLM decides add/update/delete at write. No human authority, no binding, no re-alarm.
- Zep/Graphiti: bi-temporal graph, contradicted edge gets `t_invalid`, **recency wins** → represents SUPERSEDED, never FALSE. The same wrong value can win again later.
- Cognee reweights by popularity; Anthropic memory tool = LRU; OpenAI = "guidance only." None can say *"a human decided this, and it stays decided."*

**Helicon's wedge (three things none have):**
1. **Human ruling is the authority**, not an LLM guess — the operator is the ground-truth oracle, the ruling is the held-out key.
2. **The ruling BINDS** — compiles into `GOLDEN_RULES`, enforced at write time by `guard` (section 2).
3. **Never-twice / re-alarm** — a ruled-wrong value re-alarms the instant it returns in newer memory; recency cannot overturn a ruling.

**The framing that disarms "yet another store":** Helicon runs **on** Mem0 (read-only) — Alibaba's own recommended Qwen memory backend. Not a competitor to the store the judges' stack recommends; the exam that store never runs on itself.

**The proof LLM-judging isn't enough:** every model (Qwen, Claude, GPT) missed **unit-drift** (points-as-dollars) in the bake-off. A deterministic exam catches it; an LLM grading its own context does not.
