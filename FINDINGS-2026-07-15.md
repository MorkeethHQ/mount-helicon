## 2. 🔴 The ruling loop poisons itself (a Jul-6 regression)

`resolve --list` shows **#361 Itai wedding: 08-14..08-22 vs 09-11..09-13** as a live
critical contradiction. It was already ruled on Jul 11 (#322) and the ruling is compiled
into GOLDEN_RULES. The source of the "wrong" side:

```
$ grep -n "08-14" memory/status_2026-07-11.md
16:   - #322 Itai wedding = **09-11..09-13** (the "Aug 14-22" cube was the Italy trip mislabeled)
```

That line **is the correction**. The parser lifts the dead value out of the sentence
written to kill it and files a fresh contradiction. GOLDEN_RULES already carries a
precedent for this exact class ("false positive: LOUPE correction banner parsed as the
'open' side (both-poles line, fixed Jul 6)") — so the Jul-6 fix does not cover
`X was mislabeled as Y` phrasing.

**Consequence: every ruling Oscar makes plants a future false alarm.** That inflates R1
forever and quietly discredits the whole rot exam, which is the one thing that must stay
trustworthy. Fix the both-poles parser for correction-phrasing, add #361 as a regression
test, then re-scan and see how much of R1's "9 live conflicts" evaporates.

## 3. 🟠 R4 rename rot is NOT hygiene — it blacked out a production system for 13 days

```
$ python3 -m helicon.cli rot
  R4  Supersession / rename       ROT FOUND
      RELAY->FAVOUR: 606 live dead-name ref(s) = 92 history + 190 rename-aware + 324 current-claim(s)
```

Today the FAVOUR terminal found the consequence (`world-relay/BUILD-PLAN-2026-07-15.md`):
all **107** production tasks carry `agent: null`. Cause chain: tasks post as
`poster: "agent:relay"` → strips to `agentId="relay"` → `getAgent("relay")` → no such key
in `AGENT_REGISTRY` → returns null, **silently, no error, no log**. `agent:relay` is a
dead name (RELAY→FAVOUR, renamed 2026-07-02). Eight named agents — ShelfWatch, FreshMap,
QueuePulse, PropertyCheck, DropScout, OpenClaw, Hermes, Claude Code — have never once
fired, and their differentiated `verificationPrompt` has never run.

**The finding for Helicon:** it has been reporting 324 current-claim dead names as a
count. At least one of them was load-bearing in running code. **Dead names in prose are
rot; dead names in a code path are an outage.** That distinction doesn't exist in R4
today, and adding it turns a lint into an incident detector. Concretely: cross-reference
dead names against live code paths, env vars, and configs, and rank those separately.
This is also the single best demo beat you have — "the rename checker found a
thirteen-day production blackout."

## 4. 🟡 `gold.py:131` — a memory with empty frontmatter compiles into the law as a blank rule

`feedback_ambition_past_done.md` had `name: ""` and no description. Its stored cube title
was empty. `gold.py:131` does `g["feedback"].append({"rule": title[:120], ...})`, so the
law rendered:

```
-   
  _[feedback_ambition_past_done.md]_
```

**The rule telling agents not to stop early was loading into every session as a blank
line, and nothing complained.** I repaired the data (frontmatter + the cube title) and
recompiled — the rule now reads correctly. **The code is still wrong.** Fix the compile
path (fall back to summary/content, or refuse to emit an empty rule and warn loudly),
add a regression test, then sweep: are any other rules compiling blank or truncated at
120 chars mid-word? A law with a silent hole is worse than no law, because it is trusted.

Related: re-ingest does not refresh a cube's title when only frontmatter changes (the
content hash doesn't move), so I had to UPDATE the cube directly. That's arguably a
second bug — a memory can be edited and the store won't notice.
